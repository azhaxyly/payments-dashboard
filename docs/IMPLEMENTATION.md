# Реализация по ТЗ — тезисы и решения

Короткая записка о том, как я разложил задачу на технические требования и почему принял именно
такие решения. Это «pipeline реализации по ТЗ в виде тезисов» — от бизнес-процесса к архитектуре.

## 1. Чтение ТЗ → что на самом деле проверяют

ТЗ формулируется как «дашборд оплат/проектов/актов», но это table stakes — его соберут все. Реальный
фокус отбора (прямо из текста вакансии) — **подход к бизнес-процессу + применение AI**, и отдельно:
«будет плюсом, если покажете, как мог бы быть устроен парсинг банковской выписки».

**Вывод:** дашборд делаю чисто и расширяемо, но главный акцент и большую часть инженерных усилий
кладу в **реальный AI-пайплайн `PDF выписки → структурированный JSON → БД`** с валидацией, отсевом
непроектных операций и идемпотентностью. Это и есть дифференциатор.

## 2. Бизнес-процесс → сущности

- Деньги приходят от юрлица-плательщика за этап услуги → на каждый платёж нужен закрывающий **акт**.
- Менеджер ведёт состояние акта: отправлен → подписан; просроченные подсветить.
- «Проект» в выписке фактически = юрлицо. Моделирую обе сущности (Client ↔ Project), с заделом на
  «несколько проектов на клиента», но в seed — 1:1, потому что данные так дают.

```
Client 1─* Project 1─* Payment 1─1 Act
```

## 3. Главные инженерные решения и их обоснование

| Решение | Почему |
|---|---|
| **Бизнес-логика — чистые функции** (`server/domain/`), без БД/HTTP | Тестируемость и «единственный источник истины». Статус акта и агрегаты считаются в одном месте, на бэке, и покрыты юнит-тестами против эталонных чисел. |
| **Итоги и фильтры — на бэкенде**, не на фронте | Критерий «отделять данные / логику / отображение». Фронт ничего не пересчитывает — берёт готовый `summary`. |
| **AI structured output через tool-use + Zod** | Гарантирует валидную схему ответа модели; Zod ловит то, что модель всё же испортила, и отправляет в `needsReview`, а не молча теряет. |
| **Идемпотентность через натуральный ключ** `sha1(date\|inn\|doc\|amount)` | Повторная/частичная загрузка выписки не плодит дубли — upsert. Готово к переходу с ручной загрузки на вебхук банка. |
| **MockExtractor-фолбэк без API-ключа** | Демо и CI зелёные без ключа. Это же — демонстрация «понимания ограничений и рисков»: что делать, когда внешний AI недоступен. |
| **Явная обработка рисков AI** (422 на пустой/невалидный ответ, 503 на таймаут SDK) | Навык «какая часть пайплайна упала» из требований вакансии — отражён прямо в коде, а не задним числом. |

## 4. AI-пайплайн `POST /api/ingest` — по шагам

1. PDF → текст (`unpdf`, serverless-friendly).
2. `ClaudeExtractor`: Anthropic SDK, tool-use со схемой `ExtractionResult` (Zod → JSON-schema),
   `tool_choice` форсит вызов инструмента. Модель — через env `AI_MODEL`.
3. Постобработка: Zod-валидация → дроп непроектных операций (`isProjectPayment=false`) в `skipped` →
   `confidence < 0.6` в `needsReview` → нормализация → натуральный ключ → upsert (Client/Project/Payment/Act).
4. Ответ: `{ imported, skipped, needsReview, items, extractor }` — фронт показывает отчёт распознавания.

## 5. Где это масштабируется (embeddings / RAG)

Текущая классификация этапа услуги — гибрид «правила + LLM». Прод-эволюция:

- **Embeddings-классификация `serviceStage`:** эмбеддинг назначения платежа → косинусная близость к
  эмбеддингам словаря этапов; LLM-фолбэк только на низкой уверенности. Дешевле и стабильнее.
- **RAG по договорам/счетам клиента:** при ingestion подтягивать релевантные пункты договора (vector
  store по ИНН клиента) и сопоставлять платёж с конкретным этапом работ — авто-привязка к проекту.
- **Замена ручной загрузки на вебхук/очередь банка:** тот же `ingestPdf`, идемпотентность уже готова.

Слои, быстрый старт и деплой — в [README](../README.md). Ниже — справочник: схема данных, логика
статусов и REST API.

## 6. Что осознанно НЕ делал (по ТЗ)

Бухучёт, интеграции банк/1С, авторизация/роли, ЭЦП, генерация актов, сложный дизайн. Объём держал под
тестовое — глубина в AI-части, ширина по дашборду.

# Справочник

## 7. Схема данных (ER + колонки)

```
Client (юрлицо/ИП)  1───*  Project  1───*  Payment  1───1  Act
  inn, ogrn, name,         name,           date, amount,    is_sent, sent_at,
  legal_type,              client_id,      purpose, stage,   is_signed, signed_at,
  bank_account, bank       status          invoice, doc,     manager_comment
                                           natural_key(uniq) (status вычисляется, не хранится)
```

- «Проект» в выписке фактически = юрлицо-плательщик. Моделируем обе сущности (возможны несколько
  проектов на одно юрлицо), в seed по умолчанию 1 проект на клиента — так дают исходные данные.
- 1 платёж = 1 акт (1:1).
- `natural_key = sha1(date | inn | doc | amount)` — обеспечивает **идемпотентность** AI-ingestion:
  повторная загрузка того же PDF делает upsert, а не плодит дубли.

Доменная логика — чистые функции в [`server/domain/`](../server/domain/) (без БД, без HTTP), на них
юнит-тесты: [`actStatus.ts`](../server/domain/actStatus.ts) (статус акта) и
[`aggregate.ts`](../server/domain/aggregate.ts) (сводка/итоги). Оркестрация поверх Drizzle — в
[`server/services/`](../server/services/), REST — в [`server/api/`](../server/api/). **Фильтрация и все
итоги считаются на бэкенде** — фронт их не пересчитывает.

## 8. Логика статусов акта (4 состояния + порог)

| Условие | Статус | Метка |
|---|---|---|
| не отправлен | `not_sent` | Акт не отправлен |
| отправлен, не подписан | `waiting` | Ждём подпись |
| подписан | `closed` | Закрыт |
| не закрыт и старше `ATTENTION_DAYS` (21) дней от `NOW` | `attention` | Требует внимания (оверлей поверх not_sent/waiting) |

Инвариант записи: `is_signed=true` форсит `is_sent=true`; `sent_at`/`signed_at` проставляются при
переходе флага в `true`. «Сегодня» берётся из конфигурируемой `NOW` (по умолчанию `2026-08-09`, конец
периода выписки), чтобы порог срабатывал на демо-данных.

## 9. REST API

```
GET   /api/payments?project&legalEntity&stage&from&to&sent(yes|no)&signed(yes|no)&status&q  → { items, summary }
GET   /api/projects?<те же фильтры>                                                          → { items }
GET   /api/summary?<те же фильтры>                                                           → SummaryDTO
GET   /api/filter-options                                                                    → { projects, legalEntities, stages, dateRange }
PATCH /api/acts/:id   { isSent?, isSigned?, comment? }                                       → ActDTO (с пересчитанным status)
POST  /api/ingest     (multipart: file=PDF)                                                  → { imported, skipped, needsReview, items }
```

Фронт после PATCH делает оптимистичный апдейт + рефетч (итоги всегда приходят с бэка).

## 10. Допущения и эталонные числа

- Период выписки фиксированный (16.07.2026–09.08.2026); «сегодня» — конфигурируемая `NOW`.
- «Проект» = юрлицо-плательщик; в seed 1 проект на клиента.
- AI-извлечение идемпотентно по натуральному ключу (дата+ИНН+док+сумма).

Golden-fixture / тесты агрегации: 24 оплаты · 19 уникальных юрлиц · итог **1 405 820,00 ₽** · исходно
всё не отправлено/не подписано (подписано 0 ₽, «не закрыто» = весь итог). Крупнейший плательщик —
ООО «ПЛАТФОРМА-ЛК» (266 000 ₽).

> Драйвер — `postgres-js`; доменный слой, сервисы и API зависят только от Drizzle-абстракции. Под
> высокую конкурентность на serverless имеет смысл перейти на HTTP-драйвер Neon
> (`@neondatabase/serverless` + `drizzle-orm/neon-http`) — точечная замена в
> [`db/index.ts`](../db/index.ts).
